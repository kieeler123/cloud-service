1) 기본 질문 (거의 무조건 나옴)
1-1 목적/동기
🇯🇵 Q1. なぜこのプロジェクトを作りましたか？

A.
日本企業向けのポートフォリオとして、React の基本だけでなく Firebase（Auth/Firestore/Storage）を含む実装経験を証明したかったからです。
また、将来的なデジタルノマド活動の基盤として拡張できる構造を意識しました。

🇰🇷

일본 기업 면접용 포트폴리오로 React 기본기뿐 아니라 Firebase(Auth/Firestore/Storage)까지 포함한 구현 경험을 증명하고 싶었습니다.
또한 디지털노마드 기반 프로젝트로 확장 가능한 구조를 의식했습니다.

🇯🇵 Q2. なぜ Firebase を選びましたか？

A.
個人開発でも短期間で「認証＋DB＋ストレージ」を統合して実装できるからです。
実務に近い機能（権限・アップロード・ユーザー管理）を一気に作れる点が魅力でした。

🇰🇷

개인 개발에서도 짧은 기간에 인증+DB+스토리지를 통합 구현할 수 있기 때문입니다.
권한/업로드/유저 관리 등 실무형 기능을 한 번에 만들 수 있는 점이 장점이었습니다.

1-3 역할/폴더 구조
🇯🇵 Q4. フォルダ構成はどう設計しましたか？

A.
責務分離を意識して、UI部品は components、グローバル状態は contexts、ページは pages、Firebase処理は lib にまとめました。

🇰🇷

책임 분리를 위해 UI 컴포넌트는 components, 전역 상태는 contexts, 페이지는 pages, Firebase 관련 처리는 lib로 분리했습니다.

2) 중급 질문 (차별화/실무감 강조)
2-1 설계(데이터/삭제)
🇯🇵 Q5. Trash を isTrashed フラグ方式にした理由は？

A.
復元機能を実装したかったのと、「削除」と「完全削除」を分けた方が安全で運用しやすいからです。
論理削除により、UIとデータ管理も単純化できます。

🇰🇷

복구 기능이 필요했고, 삭제와 완전삭제를 분리하는 게 안전하고 운영에도 유리하기 때문입니다.
논리 삭제로 UI/데이터 관리도 단순해집니다.

🇯🇵 Q6. 「削除」と「完全削除」の違いは？

A.
削除は isTrashed=true にする状態変更（復元可能）で、完全削除は Storage と Firestore の両方から消す別操作です。

🇰🇷

삭제는 isTrashed=true로 상태 변경(복구 가능)이고, 완전삭제는 Storage와 Firestore에서 완전히 제거하는 별도 작업입니다.

2-2 업로드/UX
🇯🇵 Q7. アップロード進捗UIを付けた理由は？

A.
アップロードには待ち時間があり、進捗が見えないと不安になるためです。
実務でもユーザー体験に直結するので重要だと考えました。

🇰🇷

업로드는 대기 시간이 생기기 때문에 진행률이 없으면 사용자가 불안해합니다.
실무에서도 UX에 직접 영향을 주는 요소라고 생각했습니다.

2-3 테마/다국어
🇯🇵 Q8. テーマ切替を Context で管理した理由は？

A.
複数ページで共通で使う状態なので、props のバケツリレーを避けるために Context を使いました。
切替の影響範囲を最小化できます。

🇰🇷

여러 페이지에서 공통으로 쓰는 상태라 prop drilling을 피하려고 Context를 사용했습니다.
변경 영향 범위를 최소화할 수 있습니다.

🇯🇵 Q9. i18n を入れた理由は？

A.
日本企業向けポートフォリオとして日本語対応が必須だったこと、将来的な海外展開を想定して設計段階から多言語対応を入れました。

🇰🇷

일본 기업 포트폴리오이므로 일본어 지원이 필요했고, 장기적으로 해외 확장까지 고려해 설계 단계부터 다국어를 넣었습니다.

3) 깊이있는 질문 (합격권 질문)
3-1 보안/권한 (일본 면접에서 핵심)
🇯🇵 Q10. セキュリティ（権限設計）はどうしていますか？

A.
Firebase Authentication を前提に Firestore / Storage の Rules でアクセス制御します。
データ取得は owner(uid) を条件にし、他ユーザーのデータは参照できない設計です。

🇰🇷

Firebase Authentication을 전제로 Firestore/Storage Rules로 접근 제어합니다.
데이터 조회는 owner(uid)를 조건으로 하고 타 유저 데이터는 접근 불가능하도록 설계했습니다.

🇯🇵 Q11. フロントだけで owner(uid) 条件を付ければ十分ですか？

A.
不十分です。フロントは改ざん可能なので、Rules側でも owner一致を必須にして二重に守ります。

🇰🇷

부족합니다. 프론트는 변조 가능하기 때문에 Rules에서도 owner 일치 조건을 강제해서 이중 방어합니다.

3-2 트레이드오프 / 확장 설계
🇯🇵 Q12. 将来的に Supabase/AWS に移行するならどうしますか？

A.
Firebase固有処理を lib に集約しているので、データ層(API/Repository)を作って差し替え可能にします。
UIから直接Firebaseを触らない構造に寄せていきます。

🇰🇷

Firebase 처리를 lib로 모아놨기 때문에 데이터 계층(API/Repository)을 만들고 교체 가능하게 확장할 계획입니다.
UI에서 Firebase를 직접 호출하지 않도록 구조를 바꿉니다.

🇯🇵 Q13. もしファイル数が増えたら、課題は？

A.
一覧取得が重くなるので、ページネーション、インデックス設計、無限スクロール等が必要です。

🇰🇷

리스트 조회가 무거워지므로 페이지네이션, 인덱스 설계, 무한 스크롤 등이 필요합니다.

3-3 실전형 운영 질문
🇯🇵 Q14. チーム開発ならまず何を改善しますか？

A.
型定義の整理、フォルダ規約、eslint/prettier、docs整備を優先します。
また、API層を作って責務分離を強化します。

🇰🇷

타입 정리, 폴더 규칙, eslint/prettier, 문서 정비를 우선합니다.
그리고 API 계층을 만들어 책임 분리를 강화합니다.

4) “Next.js 프로젝트와 비교” 질문 대응 (너에게 중요)
🇯🇵 Q15. Next.js + MongoDB版も作ったそうですが、何が違いますか？

A.
Cloud Driveは Firebase 중심으로 빠르게 제품형 기능을 구현한 프로젝트이고,
Next.js版は API/DB/배포/타입 트러블슈팅까지 포함한 풀스택 확장 경험을 보여주는 프로젝트입니다。
両方で「目的に応じた技術選択」を意識しました。

🇰🇷

Cloud Drive는 Firebase 중심으로 빠르게 제품형 기능을 구현한 프로젝트이고,
Next.js 버전은 API/DB/배포/타입 에러 해결까지 포함한 풀스택 확장 경험을 보여주는 프로젝트입니다.
둘 다 “목적에 맞춘 기술 선택”을 의식했습니다.
